DAY 5 Friday

Integration, Performance & Real Data

End-to-end integration with real data flows, performance profiling, and load testing

⚡ Performance

🔗 Full Integration Testing

  • ✦ Complete task lifecycle with a real Docker stack — create task via API → Redis queue → worker picks up → DB status updates → WebSocket notifies frontend → frontend reflects COMPLETED
  • ✦ Campaign full flow — create template → create campaign → add recipients CSV → launch → verify each recipient task created in queue → monitor send progress in real-time dashboard
  • ✦ Retry and DLQ flow — inject a failing task → verify retry attempts with backoff → verify ends in DLQ → replay from DLQ → verify completes
  • ✦ Worker drain — shutdown signal → worker completes in-flight tasks → de-registers → verify no orphaned tasks
  • ✦ Multi-user isolation — two users, each sees only their own tasks (auth scoping)

⚡ Performance Profiling

  • ✦ API benchmarking — profile all endpoints with locust or hey, identify any >100ms p95 outliers
  • ✦ DB query analysis — turn on echo=True in SQLAlchemy, capture all slow queries (>10ms), add indexes or rewrite
  • ✦ Frontend bundle audit — run vite build --analyze, identify large chunks, add dynamic imports for heavy pages
  • ✦ WebSocket load test — 100 simultaneous WS connections, verify no message loss, memory stable
  • ✦ Redis memory audit — verify no key accumulation, TTLs set on all temporary keys
  • ✦ Docker resource check — CPU/memory usage of all containers at steady state vs load
🎯 Performance Acceptance Criteria
  • ✓ GET /health responds in <10ms always
  • ✓ GET /api/v1/tasks (page 1) <50ms p95
  • ✓ POST /api/v1/tasks <80ms p95
  • ✓ Frontend initial load <2s on 3G throttle
  • ✓ WebSocket message latency <20ms event→client
  • ✓ Full JS bundle <500KB gzipped
  • ✓ 100K tasks/hr sustained throughput
  • ✓ Zero memory leak over 1hr continuous operation
